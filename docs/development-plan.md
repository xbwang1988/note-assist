## 1. 项目背景与目标

### 1.1 项目背景

在信息爆炸的时代，高效的知识管理工具成为个人和团队的刚需。市面上的笔记产品（有道云笔记、印象笔记、Notion 等）功能丰富但普遍存在以下问题：

- **重度依赖网络**：离线场景下功能受限
- **数据安全隐患**：笔记数据存储在第三方云端，用户对数据掌控力弱
- **功能冗余**：大量用户只需要轻量化的笔记能力，但产品设计过于复杂
- **本地化不足**：部分产品中文支持和本土化体验不够友好

基于以上痛点，本项目旨在开发一款**轻量级、本地优先、功能完备**的桌面笔记客户端。

### 1.2 项目目标

| 维度 | 目标描述 |
| :--- | :--- |
| 功能目标 | 覆盖日常笔记记录、整理、导出的核心需求，提供富文本 + Markdown 双编辑模式 |
| 体验目标 | 界面简洁美观，操作直觉化，冷启动 < 3 秒 |
| 技术目标 | 基于 Electron 构建跨平台桌面应用，数据本地持久化，零后端依赖 |
| 交付目标 | v1.0 完成核心功能并发布 Windows 安装包 |

## 2. 产品定位与范围

### 2.1 产品定位

**面向日常办公的智能笔记助手**——参考有道云笔记、印象笔记的核心体验，聚焦本地化、轻量化。

### 2.2 目标用户

- 日常办公人员：会议纪要、工作日志、周报撰写
- 学生群体：课堂笔记、读书笔记、学习资料整理
- 技术人员：代码片段管理、技术文档编写（Markdown）
- 个人知识管理爱好者：建立个人知识库

### 2.3 核心价值主张

| 特性 | 说明 |
| :--- | :--- |
| 本地优先 | 数据存储在本地，无网络也能正常使用 |
| 双模式编辑 | 富文本所见即所得 + Markdown 专业编辑 |
| 桌面便签 | 独立悬浮窗口，快速记录待办事项 |
| 零配置 | 开箱即用，无需注册账号 |

## 3. 技术选型与架构设计

### 3.1 技术选型

| 技术栈 | 选型 | 选型理由 |
| :--- | :--- | :--- |
| 运行时 | Electron v33 | 跨平台桌面应用首选，Web 技术栈降低开发成本，生态成熟 |
| 前端框架 | 原生 HTML/CSS/JS | 项目规模可控，避免框架引入的复杂度和包体积；对 DOM 操作有完全控制 |
| 富文本编辑 | contentEditable + execCommand | 浏览器原生支持，零依赖；满足基础富文本需求 |
| Markdown 解析 | marked.js | 轻量高性能，GFM 支持完善 |
| 代码高亮 | highlight.js | 支持 100+ 语言，主题丰富 |
| 数学公式 | KaTeX | 比 MathJax 更快的渲染速度，支持 LaTeX 子集 |
| 数据持久化 | localStorage | 零配置，适合单机场景；约 5-10MB 容量满足初期需求 |
| 打包工具 | electron-builder | 支持 NSIS 安装包，配置灵活，社区活跃 |

### 3.2 架构设计

```
┌─────────────────────────────────────────────────┐
│                 Electron 主进程                   │
│  main.js                                         │
│  ┌───────────┐ ┌───────────┐ ┌──────────────┐   │
│  │ 窗口管理   │ │ 系统托盘   │ │  菜单系统    │   │
│  └───────────┘ └───────────┘ └──────────────┘   │
│  ┌───────────┐ ┌───────────┐ ┌──────────────┐   │
│  │ IPC 通信   │ │ PDF 导出   │ │  边缘检测    │   │
│  └───────────┘ └───────────┘ └──────────────┘   │
└─────────────────────────────────────────────────┘
        │ preload.js          │ preload-sticky.js
        │ (contextBridge)     │ (contextBridge)
        ▼                     ▼
┌──────────────────┐  ┌──────────────────┐
│    主窗口渲染进程  │  │  便签窗口渲染进程  │
│  index.html       │  │  sticky.html      │
│  ┌──────────────┐ │  │  ┌──────────────┐ │
│  │ 三栏布局      │ │  │  │ Todo 管理     │ │
│  │ 侧边栏│列表│编辑│ │  │  │ 优先级/筛选   │ │
│  ├──────────────┤ │  │  │ 主题切换      │ │
│  │ Store 数据层  │ │  │  └──────────────┘ │
│  │ (localStorage)│ │  │  localStorage     │
│  └──────────────┘ │  └──────────────────┘
└──────────────────┘
```

### 3.3 安全架构

- **进程隔离**：`contextIsolation: true`，主进程与渲染进程完全隔离
- **预加载桥接**：通过 `contextBridge.exposeInMainWorld` 安全暴露有限 API
- **禁用 Node**：渲染进程 `nodeIntegration: false`，防止 XSS 攻击链
- **外链拦截**：所有外部链接通过 `shell.openExternal` 在系统浏览器打开
- **单实例锁**：`app.requestSingleInstanceLock()` 防止多实例冲突

### 3.4 第三方库管理策略

所有第三方库本地化存储在 `vendor/` 目录，不依赖 CDN，确保离线可用：

| 库 | 用途 | 文件 |
| :--- | :--- | :--- |
| marked.js | Markdown 解析 | `vendor/marked.min.js` |
| highlight.js | 代码高亮 | `vendor/highlight.min.js` |
| KaTeX | 数学公式 | `vendor/katex.min.js` + CSS + 15 个字体文件 |

## 7. 风险识别与应对

| 风险 | 影响 | 概率 | 应对策略 |
| :--- | :--- | :--- | :--- |
| localStorage 容量限制（5-10MB） | 笔记数量受限，大量图片 base64 会快速耗尽 | 高 | v2.0 迁移至 IndexedDB 或 SQLite；图片改为文件系统存储 |
| Electron 安装包体积大（~270MB） | 用户下载和分发成本高 | 中 | 使用 electron-builder NSIS 压缩；后续考虑增量更新 |
| execCommand API 已废弃 | 未来浏览器可能移除支持 | 低 | 核心功能在 Electron 内置 Chromium 中稳定；长期可迁移到 Tiptap/ProseMirror |
| 无云同步的数据安全 | 本地数据损坏或丢失无法恢复 | 中 | 提供 JSON 全量备份导出功能（已实现）；v2.0 增加云端备份 |
| Windows GPU 兼容性 | 部分环境下 GPU 渲染崩溃 | 中 | 已通过 `disableHardwareAcceleration()` 全局禁用硬件加速 |
| 第三方库安全漏洞 | 依赖库存在已知漏洞 | 低 | 本地化管理，定期检查更新；核心库（marked/highlight/katex）社区维护活跃 |
