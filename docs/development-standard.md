## 1. 产品立项规范

### 1.1 立项前置条件

在正式立项前需完成以下准备工作：

- [ ] 明确产品定位和目标用户
- [ ] 完成竞品调研（至少 3 款竞品）
- [ ] 确认技术可行性
- [ ] 初步评估开发周期和资源需求

### 1.2 PRD 模板

产品需求文档（PRD）应包含以下章节：

```markdown
# [产品名称] 产品需求文档（PRD）

## 版本记录
| 版本 | 日期 | 说明 |

## 一、产品概述
- 产品名称
- 产品定位（一句话描述）
- 目标用户画像
- 核心价值主张
- 技术栈概述
- 运行平台

## 二、功能需求
### 2.1 [功能模块 A]
| 子功能 | 优先级 | 详细描述 |
（按模块逐一列出所有功能，标注优先级 P0/P1/P2/P3）

### 2.2 [功能模块 B]
...

## 三、非功能需求
- 性能要求（启动时间、响应时间）
- 安全要求
- 兼容性要求
- 可用性要求

## 四、技术架构
- 系统架构图
- 数据模型设计
- 接口设计

## 五、里程碑计划
| 阶段 | 内容 | 预计周期 |

## 六、已知风险与约束
| 风险 | 影响 | 应对策略 |
```

### 1.3 评审要点

PRD 评审时关注以下维度：

| 评审维度 | 检查项 |
| :--- | :--- |
| 完整性 | 是否覆盖所有用户核心场景？ |
| 优先级 | P0/P1 功能是否聚焦核心价值？是否有功能蔓延的风险？ |
| 可行性 | 技术方案是否经过验证？有没有不可控的技术风险？ |
| 边界 | 产品范围是否清晰？哪些不做是否明确？ |
| 数据 | 数据模型是否合理？存储方案是否满足容量和性能需求？ |
| 度量 | 是否有可量化的验收标准？ |

> **云笔记经验**：初始 PRD 功能范围过大（包含云同步、AI、OCR 等），后期通过优先级裁剪聚焦 v1.0 核心功能。**立项时就应明确 MVP 范围**。

## 2. 技术选型规范

### 2.1 评估维度

对每个技术决策点，按以下维度评估：

| 评估维度 | 权重 | 说明 |
| :--- | :--- | :--- |
| 项目适配度 | ★★★★★ | 是否匹配项目规模、复杂度、团队技术栈 |
| 开发效率 | ★★★★ | 学习曲线、开发速度、调试便利性 |
| 生态成熟度 | ★★★★ | 社区活跃度、文档完善度、第三方库丰富度 |
| 性能表现 | ★★★ | 运行时性能、包体积、内存占用 |
| 可维护性 | ★★★ | 代码组织、模块化支持、长期维护成本 |
| 扩展性 | ★★ | 未来功能扩展的灵活度 |

### 2.2 决策矩阵模板

```markdown
## 技术决策：[决策主题]

### 背景
[为什么需要做这个决策]

### 候选方案

| 维度 | 方案 A | 方案 B | 方案 C |
| :--- | :--- | :--- | :--- |
| 方案描述 | ... | ... | ... |
| 项目适配度 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 开发效率 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 生态成熟度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| 性能表现 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
```

### 2.3 常见技术选型参考

#### 桌面应用框架

| 框架 | 适用场景 | 包体积 | 技术栈 |
| :--- | :--- | :--- | :--- |
| Electron | Web 技术栈团队、功能复杂、跨平台 | ~270MB | HTML/CSS/JS |
| Tauri | 追求小体积、性能敏感 | ~10MB | Rust + Web |
| Flutter | 多端统一（含移动端） | ~50MB | Dart |
| .NET MAUI | 微软生态、企业应用 | ~30MB | C# |

#### 前端框架

| 场景 | 推荐 |
| :--- | :--- |
| 小型项目（< 3000 行逻辑） | 原生 JS（避免过度工程化） |
| 中型项目（3000-20000 行） | Vue 3 / React（提升开发效率） |
| 大型项目（> 20000 行） | React + TypeScript（类型安全、生态丰富） |

#### 数据存储

| 方案 | 容量 | 适用场景 |
| :--- | :--- | :--- |
| localStorage | 5-10MB | 原型验证、设置存储 |
| IndexedDB | 数百 MB | 本地数据密集型应用 |
| SQLite (better-sqlite3) | 无限制 | 结构化数据、复杂查询 |
| 文件系统 | 无限制 | 大文件、图片、附件 |

> **云笔记经验**：选用 localStorage 快速验证产品可行性是正确的，但应更早规划数据层迁移路径。建议在 PRD 中标注数据层的 "当前方案" 和 "目标方案"。

## 3. 开发流程规范

### 3.1 分支策略

推荐使用简化版 Git Flow：

```
main        ─────●─────────────●───────── 稳定发布
              ↑                   ↑
develop     ──●──●──●──●──●──●──●──── 开发集成
              ↑     ↑     ↑
feature/*   ──●──●  ●──●  ●──●     功能分支
              ↑
hotfix/*    ──●──●                  紧急修复
```

| 分支 | 用途 | 命名规范 |
| :--- | :--- | :--- |
| `main` | 稳定发布版本 | - |
| `develop` | 开发集成分支 | - |
| `feature/*` | 功能开发 | `feature/add-markdown-editor` |
| `hotfix/*` | 紧急修复 | `hotfix/fix-pdf-export` |
| `release/*` | 发布准备 | `release/v1.0.0` |

> **单人开发简化**：如果是单人开发，可以简化为 `main` + `feature/*`，无需 `develop` 分支。

### 3.2 代码规范

#### 命名规范

| 类型 | 规范 | 示例 |
| :--- | :--- | :--- |
| 文件名 | 小写 + 连字符 | `app.js`, `sticky-note.js` |
| CSS 类名 | 小写 + 连字符（BEM 可选） | `note-card`, `sidebar__header` |
| JS 变量/函数 | camelCase | `currentFilter`, `addTodo()` |
| JS 类名 | PascalCase | `Store`, `App` |
| JS 常量 | UPPER_SNAKE_CASE | `STORAGE_KEY`, `EDGE_THRESHOLD` |
| CSS 变量 | 小写 + 连字符 | `--bg-primary`, `--text-color` |

#### 代码组织

```javascript
// 推荐的文件结构顺序

// 1. 常量定义
const STORAGE_KEY = 'app_data';
const MAX_ITEMS = 100;

// 2. 全局状态
let currentState = null;

// 3. 初始化
document.addEventListener('DOMContentLoaded', () => { init(); });

// 4. 核心逻辑（按功能分组，用注释分隔）
// ========== 数据操作 ==========
function loadData() { ... }
function saveData() { ... }

// ========== UI 渲染 ==========
function render() { ... }
function updateStats() { ... }

// ========== 事件处理 ==========
function bindEvents() { ... }

// ========== 工具函数 ==========
function escapeHtml(str) { ... }
```

#### 代码质量检查清单

- [ ] 没有硬编码的魔法数字（提取为常量）
- [ ] 用户输入已做 XSS 防护（`escapeHtml`）
- [ ] 异常场景有 `try-catch` 保护
- [ ] DOM 查询结果在使用前检查是否为 null
- [ ] 事件监听器在不需要时移除（防止内存泄漏）

### 3.3 提交规范

采用 Conventional Commits 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

| type | 说明 | 示例 |
| :--- | :--- | :--- |
| `feat` | 新功能 | `feat(editor): add markdown preview mode` |
| `fix` | Bug 修复 | `fix(sticky): resolve edge hide flicker` |
| `docs` | 文档变更 | `docs: update PRD to v2.0` |
| `style` | 样式调整（不影响逻辑） | `style(sidebar): adjust tag cloud spacing` |
| `refactor` | 重构 | `refactor(store): extract notebook methods` |
| `perf` | 性能优化 | `perf(search): add plainText field for faster search` |
| `chore` | 构建/工具变更 | `chore: update electron-builder config` |

### 3.4 渐进式开发原则

从云笔记项目总结出的开发节奏：

1. **先跑通，再完善**：每个功能先实现最小可用版本（MVP），验证通过后再打磨细节
2. **先核心，再周边**：编辑器 → 管理 → 效率工具 → 导出 → UI 优化
3. **先单机，再联网**：本地功能全部就绪后再考虑云端能力
4. **先功能，再性能**：功能正确优先于性能优化

```
MVP → 核心功能 → 完整功能 → 优化打磨 → 打包发布
 ↑      ↑          ↑          ↑           ↑
 每一步都应该是可运行、可验证的
```

## 4. 测试规范

### 4.1 测试类型

| 测试类型 | 负责人 | 工具/方法 | 覆盖范围 |
| :--- | :--- | :--- | :--- |
| 单元测试 | 开发 | Jest / Vitest | 数据层（Store 类）、工具函数 |
| 集成测试 | 开发 | Playwright / Spectron | 多模块交互、IPC 通信 |
| E2E 测试 | 测试 | Playwright | 用户核心流程 |
| 手动测试 | 测试 | 测试用例表 | 全功能覆盖 |
| 兼容性测试 | 测试 | 多环境运行 | Windows 10/11，不同分辨率 |

### 4.2 核心测试用例模板

```markdown
## 测试用例：[功能名称]

### 前置条件
- [测试前需要的状态]

### 测试步骤
1. [操作步骤 1]
2. [操作步骤 2]
3. ...

### 预期结果
- [期望的行为]

### 实际结果
- [ ] 通过
- [ ] 不通过：[描述实际行为]

### 备注
- [特殊说明]
```

### 4.3 验收标准模板

```markdown
## 功能验收：[版本号]

### 功能完整性
- [ ] [功能 A] 正常工作
- [ ] [功能 B] 正常工作
- [ ] ...

### 性能指标
- [ ] 冷启动时间 < [X] 秒
- [ ] 核心操作响应 < [X] ms
- [ ] 内存占用 < [X] MB

### 兼容性
- [ ] Windows 10 正常运行
- [ ] Windows 11 正常运行
- [ ] 高 DPI 显示正常

### 安全性
- [ ] 无已知安全漏洞
- [ ] 进程隔离正常
- [ ] 输入过滤有效
```

> **云笔记经验**：v1.0 缺少自动化测试是主要不足。建议至少为数据层（Store）和工具函数编写单元测试。

## 5. 打包发布规范

### 5.1 构建流程

```
代码冻结 → 版本号更新 → 构建打包 → 安装测试 → 发布
```

#### 5.1.1 版本号规范

采用语义化版本（Semantic Versioning）：

```
主版本号.次版本号.修订号
  X    .   Y   .   Z

X: 不兼容的 API 变更
Y: 向下兼容的功能新增
Z: 向下兼容的问题修复
```

| 场景 | 版本变化 | 示例 |
| :--- | :--- | :--- |
| 重大功能发布 | X+1.0.0 | 1.0.0 → 2.0.0 |
| 新增功能 | X.Y+1.0 | 1.0.0 → 1.1.0 |
| Bug 修复 | X.Y.Z+1 | 1.0.0 → 1.0.1 |
| 预发布 | X.Y.Z-beta.N | 2.0.0-beta.1 |

#### 5.1.2 构建前检查清单

- [ ] 所有功能测试通过
- [ ] `package.json` 中版本号已更新
- [ ] 关闭所有开发调试代码（console.log 等）
- [ ] 确认无进程占用打包目标目录
- [ ] 第三方库版本锁定（无未经验证的更新）

#### 5.1.3 Electron 打包配置参考

```json
{
  "build": {
    "appId": "com.company.product",
    "productName": "产品名称",
    "directories": {
      "output": "dist"
    },
    "files": [
      "main/**/*",
      "preload/**/*",
      "renderer/**/*",
      "sticky/**/*",
      "vendor/**/*",
      "assets/**/*"
    ],
    "win": {
      "target": [
        { "target": "nsis", "arch": ["x64"] }
      ]
    },
    "nsis": {
      "oneClick": false,
      "allowToChangeInstallationDirectory": true,
      "createDesktopShortcut": true,
      "createStartMenuShortcut": true,
      "shortcutName": "产品名称"
    }
  }
}
```

### 5.2 发布检查清单

- [ ] 安装包可正确安装和卸载
- [ ] 桌面快捷方式和开始菜单正常创建
- [ ] 应用首次启动正常（无崩溃、无白屏）
- [ ] 核心功能在全新环境下正常工作
- [ ] 便携版解压即用

### 5.3 常见打包问题及解决

| 问题 | 原因 | 解决方案 |
| :--- | :--- | :--- |
| 打包失败：文件被占用 | Electron 开发进程未退出 | kill 掉 Electron 进程再打包 |
| 打包后资源加载失败 | 文件路径未包含在 `files` 配置中 | 检查并补充 `files` 数组 |
| 安装后应用白屏 | 硬件加速不兼容 | 禁用硬件加速 |
| 图标不显示 | 图标格式不支持或路径错误 | 使用 PNG/ICO 格式，检查路径 |
| 中文乱码 | NSIS 语言包未配置 | 配置 `installerLanguages` 和 `language` |

## 6. 文档规范

### 6.1 项目需要产出的文档

| 文档 | 阶段 | 负责人 | 说明 |
| :--- | :--- | :--- | :--- |
| 产品需求文档（PRD） | 立项阶段 | 产品经理 | 功能需求、优先级、验收标准 |
| 技术方案文档 | 设计阶段 | 技术负责人 | 架构设计、技术选型、数据模型 |
| 产品开发计划书 | 规划阶段 | 项目经理 | 里程碑、资源、风险 |
| 开发工作流记录 | 开发全程 | 开发团队 | 各阶段工作内容、问题、决策 |
| 测试报告 | 测试阶段 | 测试工程师 | 测试用例执行结果、Bug 列表 |
| 发布说明 | 发布阶段 | 项目经理 | 版本功能列表、已知问题 |
| 项目复盘报告 | 收尾阶段 | 项目经理 | 做得好的、待改进的、经验总结 |

### 6.2 文档编写规范

#### 通用规则

- 使用 Markdown 格式，确保在编辑器和 GitHub 上均可正确渲染
- 文件命名使用小写 + 连字符：`development-plan.md`
- 每个文档开头标注版本号、更新日期、作者
- 使用表格呈现结构化信息，便于快速扫描
- 代码块标注语言类型，便于高亮

#### 文档模板 — 发布说明

```markdown
# [产品名称] v[版本号] 发布说明

发布日期：[YYYY-MM-DD]

## 新增功能
- [功能 1 描述]
- [功能 2 描述]

## 优化改进
- [优化 1 描述]
- [优化 2 描述]

## Bug 修复
- [Bug 1 描述]
- [Bug 2 描述]

## 已知问题
- [问题 1 描述及临时解决方案]

## 升级说明
- [是否需要数据迁移]
- [兼容性注意事项]
```

## 7. 项目复盘规范

| 时机 | 参与者 | 时长 |

| 版本发布后 | 全体团队 | 60-90 分钟 |
| 重大 Bug 修复后 | 相关开发/测试 | 30 分钟 |

### 7.2 复盘模板

```markdown
# [版本号/里程碑] 项目复盘报告
```

### 7.3 经验沉淀机制

复盘中提炼出的可复用经验，应沉淀到以下位置：

| 经验类型 | 沉淀位置 | 示例 |
| :--- | :--- | :--- |

### 7.4 云笔记项目复盘示例

#### 做得好的

| 项目 | 详情 |
| :--- | :--- |
| 渐进式开发 | 每步可验证，避免了大规模返工 |
| 技术选型克制 | 不用框架、不用构建工具，保持项目简洁可控 |
| 本地化策略 | 第三方库全部本地化，杜绝网络依赖 |
| 安全优先 | 从第一天起采用 contextIsolation 安全模式 |
| PRD 先行 | 先写需求文档再开发，功能有据可循 |

#### 需要改进的

| 项目 | 详情 | 改进方案 |
| :--- | :--- | :--- |
| 数据层 | localStorage 容量限制成为硬伤 | 立项时规划数据迁移路径 |
| 测试覆盖 | 零自动化测试，回归靠手动 | 至少为核心数据层写单元测试 |
| 代码模块化 | 单文件 2100 行，后期维护成本高 | > 500 行即考虑拆分 |
| CSS 架构 | 单文件 1800 行无模块化 | 采用 CSS Modules 或按组件拆分 |
| 图片存储 | base64 方案不可持续 | 设计阶段即规划文件系统存储 |
